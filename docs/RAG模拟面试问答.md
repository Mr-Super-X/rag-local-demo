# RAG 技术面试模拟问答

> 涵盖 RAG 基础原理、工程实现、生产优化三个层次。适合准备 AI 工程师 / RAG 开发岗位面试时自测。
> 
> **面试前建议先通读**：**[理解 RAG：从概念到实践](理解RAG-从概念到实践.md)**（15 分钟）建立完整的 RAG 概念框架，再来刷题效果更好。

---

## 基础概念

### Q1: 请用一句话解释什么是 RAG？

**A:** RAG（Retrieval-Augmented Generation，检索增强生成）是一种在 LLM 生成回答之前，先从外部知识库中检索相关信息作为上下文的技术方案，目的是让 LLM 能回答训练数据之外的问题，并减少幻觉。

---

### Q2: RAG 解决的核心问题是什么？

**A:** 三个核心痛点：

1. **知识截止**：LLM 的训练数据有截止日期，无法回答训练之后发生的事
2. **幻觉**：没有知识来源时，LLM 倾向于编造——RAG 给了它"参考资料"来约束输出
3. **私有知识**：企业的内部文档、代码库、合同等数据 LLM 从未见过，RAG 让 LLM 能"读懂"这些专属数据

---

### Q3: RAG 的基本工作流程是怎样的？

**A:** 分为两个阶段。详见 [理解 RAG §1.3](理解RAG-从概念到实践.md) 的数据流图和 §3.1 的六步流程。

**离线阶段（索引）：**
```
原始文档 → 解析提取文本 → 切分为 Chunk → Embedding 向量化 → 存入向量数据库
```

**在线阶段（查询）：**
```
用户问题 → Embedding 向量化 → 向量数据库相似度搜索 → 取 Top-K 相关 Chunk
→ Prompt 模板组装（system prompt + context + question）→ LLM 生成回答
```

---

## 文档处理

### Q4: 为什么要做文档切分（Chunking）？不切直接整篇检索不行吗？

**A:** 两个原因：

1. **Embedding 模型有输入长度限制**——比如 all-MiniLM-L6-v2 最多处理 512 个 token，超长的文档根本塞不进去
2. **检索精度问题**——整篇文档的 embedding 是一个平均语义，具体到某个细节问题时匹配度很低。切成小块后每块的语义更聚焦，匹配精度更高

---

### Q5: 常见的切分策略有哪些？各有什么优劣？

**A:**

| 策略 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| 固定长度切分 | 按字符/token 数硬切 | 简单、快 | 可能切断句子或段落，语义不完整 |
| 递归字符切分 | 按分隔符优先级递进切分（`\n\n` → `\n` → `。` → 空格 → 字符） | 尽量保持语义边界 | 实现稍复杂 |
| 语义切分 | 用模型判断"此处是否该断" | 语义边界最准确 | 慢、需要额外的模型 |
| 句子级切分 | 按句号/感叹号/问号切 | 每块是一个完整句子 | 块大小不均匀 |

**实际项目中常用递归字符切分作为默认方案**——它在简单和效果之间取得了好的平衡。

---

### Q6: Chunk Size 和 Chunk Overlap 怎么选？

**A:**

- **Chunk Size**：太小（< 256 token）→ 信息碎片化，LLM 看不全上下文；太大（> 1024 token）→ 检索精度下降，一个 chunk 里混了太多不相关内容。**中文场景推荐 512 token 左右。**
- **Chunk Overlap**：相邻 chunk 之间的重叠部分，防止关键信息恰好被切在边界上。**通常设为 chunk size 的 10~20%。**

> 一个常见调试场景：用户反馈"AI 有时候能找到答案，有时候找不到"。通常是因为关键信息正好被切到了两个 chunk 的边界——一个 chunk 有前半句、另一个有后半句，单独看都不完整。**增加 overlap 可以缓解这个问题。**

---

### Q7: 扫描版 PDF（图片型）怎么处理？

**A:** 扫描版 PDF 本质是图片，没有文字层，普通解析器（pdf-parse）读不出来。需要走 OCR 流水线：

```
扫描件 → OCR 识别（PaddleOCR / Tesseract）→ 提取文字 → 进入正常切分流程
```

生产环境中建议用 Unstructured.io 或 MinerU 这类专业文档解析工具，它们内置了 OCR 能力，能自动判断页面是"文字型"还是"图片型"并走不同处理路径。

## Embedding 与检索

### Q8: Embedding 模型怎么选？维度越高越好吗？

**A:** 不完全是。维度只是一方面，还要看模型的训练数据和评测表现。

选型考量：

| 因素 | 说明 |
|------|------|
| 语言 | 中文场景优先 BGE 系列（BAAI），多语言选 multilingual-e5 |
| 维度 | 384 维够用（Demo），1024 维更精准（生产）。维度越高检索越慢、存储越大 |
| 模型大小 | 直接影响加载速度和显存/内存占用 |
| MTEB 排行榜 | 参考 [MTEB Leaderboard](https://huggingface.co/spaces/mteb/leaderboard) 上的检索任务得分 |

> 一个关键约束：**一旦换了 embedding 模型，所有存量文档必须重新入库**。因为新旧模型的向量语义空间不兼容。所以选模型先做充分测试，定了就不要轻易换。

---

### Q9: 向量相似度有哪几种计算方式？怎么选？

**A:**

| 方式 | 公式思路 | 适用场景 |
|------|---------|---------|
| 余弦相似度 | 两个向量夹角的余弦值 | **最常用**，对向量长度不敏感，适合文本语义 |
| 欧氏距离 | 空间中两点直线距离 | 向量归一化后等价于余弦 |
| 点积 | 向量逐元素相乘求和 | 某些模型（如 OpenAI embedding）专门优化过点积 |

**生产环境几乎都用余弦相似度**。如果你用的 embedding 模型已经做了 L2 归一化（如 all-MiniLM-L6-v2 设了 `normalize: true`），那余弦和欧氏距离等价。

---

### Q10: 检索结果太少或太多怎么办？

**A:**

- **检索太多不相关结果** → 提高相似度阈值（`MIN_SIMILARITY`），比如从 0.5 提到 0.6
- **检索不到结果** → 降低阈值、增加 Top-K、检查 embedding 模型是否正确加载
- **检索到了但质量差** → 加 Reranker 做精排（见 Q13）

---

## 向量数据库

### Q11: 向量数据库和传统数据库有什么区别？

**A:** 传统数据库（MySQL、PostgreSQL）擅长精确匹配——`WHERE name = '张三'`。向量数据库擅长**近似匹配**——"找和这段话最相似的内容"。

核心差异：

| | 传统数据库 | 向量数据库 |
|---|----------|----------|
| 查询方式 | 精确匹配 / 范围查询 | 向量相似度搜索（ANN） |
| 索引结构 | B-Tree / Hash | HNSW / IVF / PQ |
| 返回结果 | 精确结果 | 近似结果（精度 vs 速度可调） |

---

### Q12: 怎么选向量数据库？

**A:** 决策链：

```
规模？
├── < 10 万条、单机 → LanceDB / ChromaDB（零运维，五分钟上线）
├── 10 万~千万条 → Qdrant（单机高性能，Rust 实现） / Weaviate
├── 亿级、分布式 → Milvus / Elasticsearch 8.x+
└── 已有 ES 基础设施 → 直接 ES 加向量搜索，不用引入新组件
```

关键考量：
- **是否需要混合检索**（向量 + 关键词）→ Milvus / Weaviate
- **是否需要多租户**（不同用户/部门数据隔离）→ Milvus / Qdrant
- **运维能力** → LanceDB 零运维，Milvus 需要专门的集群管理

## 高级检索策略

### Q13: 什么是 Reranker？为什么要用它？

**A:** Reranker（重排序器）是对向量检索结果进行"精排"的模型。

**为什么要用**：向量模型（Embedding）是"双塔架构"——query 和 document 分开编码再算相似度，速度快但精度有限。Reranker（Cross-Encoder）是把 query 和 document **拼在一起**打分，精度更高但速度慢。

**用法**：向量检索召回 Top-20（粗排）→ Reranker 精排 → 取 Top-5（最终结果）。相当于"海选 → 面试"。

常用模型：BGE-Reranker-v2-m3、Cohere Rerank。

---

### Q14: 什么是混合检索（Hybrid Search）？

**A:** 同时使用**稠密向量检索（语义）**和**稀疏向量/关键词检索（精确匹配）**，合并结果。

**场景**：用户搜"合同编号 C-2025-00318"。向量检索可能找不到（这个编号太独特），但关键词检索（BM25）能精确匹配到。反过来，"怎么优雅地拒绝加班"更适合向量检索（不依赖关键词）。

合并策略：RRF（Reciprocal Rank Fusion）是常用方法——同时跑两路检索，按排名融合结果。

---

### Q15: 什么是小→大文档索引（Small-to-Big Retrieval）？

**A:** 解决一个矛盾——chunk 太小丢上下文，chunk 太大检索不精准。

**做法**：用小块做检索（精准匹配），用大块喂 LLM（充足上下文）。

实现方式：
- **父子文档**：索引时存小 chunk，每个小 chunk 关联它的"父文档"（完整段落/章节）。检索到小 chunk 后，取对应的父文档喂给 LLM
- **句子窗口**：检索到目标句子后，向前后各扩展 N 句作为上下文

## Prompt 工程

### Q16: RAG 的 Prompt 模板应该怎么写？

**A:** 一个典型的 RAG Prompt 模板包括三个要素：

1. **角色指令**（System Prompt）："你是一个基于文档检索的助手"
2. **约束规则**："仅使用提供的参考资料回答问题。如果资料中没有相关信息，请如实说明不要编造。"
3. **结构化输入**：明确标注"参考资料"和"问题"的分界

```
System: 你是一个文档问答助手。请仅使用下面提供的参考资料。
如果资料不足，请直接说"根据现有资料无法回答"。
不要编造事实，不要使用你自己的知识。

User:
参考资料：
[1] {chunk_1}
[2] {chunk_2}
...

问题：{question}

回答：
```

**关键**：**必须强调"不要编造"**。0.5B 这种小模型如果没有这个约束，非常容易在信息不足时胡编。大模型（7B+）也存在这个倾向，约束越明确越好。

---

### Q17: 如何处理"检索不到相关信息"的情况？

**A:** 三层防御：

1. **检索层**：设相似度阈值，低于阈值的 chunk 直接丢弃（如 MIN_SIMILARITY = 0.5）
2. **Prompt 层**：在 system prompt 中明确要求"不知道就说不知道"
3. **后处理层**：如果所有 chunk 都被过滤了，直接返回"未找到相关文档段落"，不调 LLM
4. **可选——LLM 自我判断**：让 LLM 在回答前先输出一个"相关性评分"，如果它觉得资料不相关就不回答

## 评估体系

### Q18: 怎么评估一个 RAG 系统的质量？

**A:** 分层评估：

| 层级 | 指标 | 衡量什么 | 怎么算 |
|------|------|---------|--------|
| 检索 | Hit Rate | Top-K 结果里有没有正确答案 | 命中的查询数 / 总查询数 |
| 检索 | MRR | 正确答案排在第几位 | 1/排名 的均值 |
| 检索 | NDCG@10 | 前 10 个结果的排序质量 | 考虑位置权重的累积增益 |
| 生成 | 忠实度 (Faithfulness) | LLM 回答是否基于检索到的 context | 将回答中的断言逐一与 context 比对 |
| 生成 | 答案相关性 | 回答是否解决了用户的问题 | 人工评分或 LLM-as-Judge |
| 端到端 | 幻觉率 | LLM 是否编造了 context 中没有的信息 | 人工标注 |

**怎么建评估数据集**：准备 100-500 个真实用户问题，人工标注每个问题的"正确答案在哪个 chunk"。跑一遍检索，看标注的 chunk 是否在检索结果里。

---

### Q19: 什么是 LLM-as-Judge？

**A:** 用一个强大的 LLM（如 GPT-4 或 Claude）来评判另一个 LLM 的输出质量。

典型用法：
```
请评估以下 RAG 系统对用户问题的回答质量。
评估维度：
1. 忠实度 (1-5): 回答是否严格基于提供的参考资料？
2. 完整性 (1-5): 回答是否完整解答了用户问题？
3. 简洁度 (1-5): 回答是否简洁，有没有冗余？

[参考资料] ...
[用户问题] ...
[RAG 回答] ...
```

这比人工评估快得多，但需要注意 LLM-as-Judge 本身也有偏见（偏好较长回答、偏好自己的输出等）。

## 生产实战

### Q20: RAG 系统上线后，最常见的故障是什么？

**A:** 按频率排序：

1. **检索质量下降**（~50%）：加了新文档后，热门查询的检索结果变差。"知识库越多越聪明"其实是错的——嘈杂数据会稀释相关结果
2. **Embedding 模型升级导致效果异常**（~20%）：换了模型但存量向量没重建
3. **LLM 幻觉增加**（~15%）：Prompt 被改了但没有回归测试
4. **性能瓶颈**（~10%）：知识库变大后，检索延迟超预算
5. **文档解析失败**（~5%）：用户上传了特殊格式的 PDF

**排查顺序**：检索 → 文档解析 → LLM。70% 的问题出在检索环节。

---

### Q21: 实时更新的知识库怎么设计？（文档增删改）

**A:**

**增量添加（最常见）**：新文档 → 解析 → 切分 → embedding → 直接插入向量库。LanceDB、Milvus 都支持增量写入。

**删除**：删除该文档的所有 chunk。如果向量库不支持按 docId 批量删除，需要遍历所有 chunk 逐个删。

**修改**：**没有"直接修改"**。正确做法是删除旧版本的所有 chunk → 重新入库新版本。

**一致性保证**：用消息队列异步处理文档变更。用户上传文件 → 消息入队 → Worker 异步处理 → 处理完成标记为可检索。大文档处理可能需要 1-2 分钟，不能让用户同步等待。

---

### Q22: 多用户/多部门的知识库如何做数据隔离？

**A:** 两种方案：

| 方案 | 做法 | 适用 |
|------|------|------|
| 物理隔离 | 每个租户独立的向量库/表 | 数据量小、租户少 |
| 逻辑隔离 | 共用向量库，每条 chunk 加 `tenantId` 字段，检索时加 `WHERE tenantId='xxx'` 过滤 | 租户多、灵活 |

逻辑隔离对向量库的要求较高——需要支持高效的标量过滤（在向量检索之前或之后做预过滤）。Milvus 的 Partition Key、Qdrant 的 Payload 过滤都支持。

**安全注意**：如果 A 部门的文档 chunk 被 B 部门的检索命中（过滤失败），就发生了数据泄露。需要在上线前做充分的隔离测试。

---

### Q23: 怎么控制 RAG 系统的成本？

**A:**

| 环节 | 省钱策略 |
|------|---------|
| Embedding | 用小模型（如 BGE-small，384 维），效果够用成本低；批量 embedding，不要逐条调 |
| 向量存储 | 用 PQ（乘积量化）压缩向量，牺牲少量精度换取 4-8x 存储缩减 |
| LLM 推理 | 缓存常见问题的回答（语义相似的问题直接返回缓存）；用 7B 模型处理大部分请求，14B 处理复杂问题（模型路由） |
| 文档处理 | OCR 只对扫描件做，文字型 PDF 不要走 OCR；去重（相同文件不重复入库） |

> 一个经验值：**检索成本通常不到生成成本的 5%。** 省钱的重点在 LLM 调用，不是检索。

### Q24: 你在这个 RAG Demo 项目中解决过哪些实际问题？

**A:** （结合本项目真实经历）

1. **npm 依赖安装失败**：`sharp` 原生模块需从 GitHub 下载，国内网络超时。解决方案——分两步安装：`npm install --ignore-scripts` + `npm rebuild`
2. **系统代理干扰网络请求**：Windows 系统级 `HTTPS_PROXY` 环境变量来自注册表，`process.env` 上的 `delete` 无法真正移除。解决方案——用 `process.env[v] = ''` 显式置空覆盖
3. **HuggingFace 域名不可达**：切换国内镜像 `hf-mirror.com`，加上 `ETIMEDOUT` 自动重试 3 次机制（用 `https.get({ agent: false })` 绕过代理直连下载）
4. **GGUF 模型文件下载损坏**：下载脚本加魔数校验（`0x47 0x47 0x55 0x46` = "GGUF"），校验不通过自动删除重下
5. **Embedding 模型下载走代理超时**：`undici.fetch` 连接超时仅 10 秒——改从本地预下载文件加载，`env.allowLocalModels = true` + `env.localModelPath`
6. **LanceDB 空表无 schema 报 500**：创建表时插入占位行定义 schema 再删除
7. **前端 JS 文件混入 TypeScript 语法**：`.js` 文件不能有 `: Type` 类型标注和 `!` 非空断言
8. **上传文件的 MIME 识别不准**：Windows 对 `.md` 文件发送 `application/octet-stream`——改按文件扩展名判断

## 开放性问题

### Q25: 如果让你设计一个企业级 RAG 知识中台，你会怎么设计架构？

**A:** 核心设计思路：**解耦 + 异步 + 可评估**。

```
                    ┌──────────┐
                    │ API 网关  │  ← 鉴权、限流
                    └────┬─────┘
                         │
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
    ┌──────────┐  ┌──────────┐  ┌──────────┐
    │ 上传服务  │  │ 检索服务  │  │ LLM 服务  │
    └────┬─────┘  └────┬─────┘  └────┬─────┘
         │              │              │
         ▼              ▼              ▼
    ┌─────────┐  ┌──────────┐  ┌──────────┐
    │ MinIO   │  │ Milvus   │  │ vLLM     │
    │ 对象存储 │  │ 向量库   │  │ GPU 推理  │
    └─────────┘  └──────────┘  └──────────┘
         │
         ▼
    ┌─────────┐
    │ 消息队列 │  ← 异步处理大文档
    │  Redis   │
    └─────────┘
```

关键设计决策：

- **上传和入库异步解耦**——上传即返回，后台 worker 慢慢处理大文档。用户不需要等 2 分钟
- **检索服务无状态**——可以水平扩展，加机器就加吞吐
- **LLM 服务独立部署**——与 GPU 强绑定，可以单独升级模型版本
- **评估管线独立**——每周自动跑一遍评估集，Hit Rate 下降就告警
- **监控覆盖四层**：检索（QPS、P99、Hit Rate）→ LLM（首 Token 延迟、吞吐）→ 上传（成功率、平均处理时长）→ 基础设施（Milvus 内存、MinIO 存储趋势）

### Q26: 你如何看待 RAG、Fine-tuning 和长上下文的关系？

**A:** 三者不互斥，是不同维度的工具：

| 方式 | 适合什么 | 不适合什么 |
|------|---------|-----------|
| **RAG** | 知识频繁变化、需要溯源、数据安全要求 | 需深度理解文档整体结构（如法律条文推理） |
| **Fine-tuning** | 让 LLM 学会特定风格/格式/领域术语 | 知识的准确性（微调是学模式，不是学事实） |
| **长上下文** | 需要完整理解单篇文档 | 成本高、速度慢、多文档场景下力不从心 |

**实际产品中通常是组合使用**：
- RAG 负责"找到相关信息"——外部知识
- Fine-tuning 负责"学会怎么回答"——风格和格式
- 长上下文负责"不漏细节"——对检索到的 chunk 做完整理解

### Q27: 你觉得 RAG 的未来发展方向是什么？

**A:** 几个趋势：

1. **Agentic RAG**：LLM 不再一步到位回答，而是多步推理——先规划需要哪些信息、调用检索工具、评估检索结果、信息不够就换个方向再搜，最后综合输出。类似人类查资料的过程
2. **多模态 RAG**：不只是文字，还能检索图片、表格、音视频。用户上传一张架构图问"这个系统的瓶颈在哪"
3. **Graph RAG**：在向量检索之外，用知识图谱组织实体关系。比如"张三的上司的下属提交了哪些合同"——这种多跳推理纯向量检索做不好
4. **Self-RAG**：LLM 在生成过程中自我评估——"这段检索到的内容真的相关吗？"——不相关就重新检索
5. **评估自动化**：用 LLM 自动生成评估数据集（问题+标准答案），持续监控 RAG 系统质量
